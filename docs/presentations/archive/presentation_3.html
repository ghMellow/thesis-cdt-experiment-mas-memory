Overview

Cos'è questo progetto

  • Un esperimento controllato multi-agente su scenari 5G, orchestrato con LangGraph
  • Due ruoli LLM — expert (senior 5G engineer) e beginner (junior) — rispondono agli stessi task
  • Due setup (1A: stesso modello per entrambi; 1B: modelli diversi per ruolo) isolano l'effetto del prompt dall'effetto del modello
  • Ogni task viene ripetuto più volte a temperatura 0.3 per misurare consistenza e varianza reale
  • Valutazione automatica: deterministica per task math, judge LLM + rubrica per task testuali

Architettura

Mappa del Codice

  • main.py — entrypoint CLI; itera esperimenti/ruoli/task/ripetizioni; genera report
  • config.py — modelli, temperature (0.3), retry, ripetizioni, soglie, TASK_MODEL_OVERRIDES
  • utils/experiment_utils.py — stato e grafo LangGraph; nodi load_task → run_agent → check_answer → save_result
  • agents/agent_runner.py — chiamata LLM via Ollama + parsing Markdown (fallback JSON); log token in/out
  • agents/judge_agent.py — judge LLM + parsing + semantic equivalence check (run_semantic_equivalence_check)
  • agents/_llm_utils.py — helper condivisi (spinner, parsing, error handling)
  • utils/evaluation_utils.py — aggregazione risultati, Brier score, consistency check a due fasi, report anomaly-focused
  • utils/task_utils.py — parsing metadata, lettura _sol.md, skip run già completate

Il Grafo LangGraph

Parametri di Esecuzione

Ripetizioni e Retry

Due meccanismi distinti con scopi ortogonali
  • REPETITIONS — ripete lo stesso task N volte; con temperatura 0.3 misura varianza LLM reale tra esecuzioni indipendenti
  • MAX_RETRIES — tentativi aggiuntivi dentro una singola ripetizione, attivati solo se il verdict è wrong
  • Retry con segnale parziale: al retry l'agente sa che la risposta precedente era wrong — sa di aver sbagliato, ma non riceve la ground truth né il feedback del judge; sa solo che il suo ragionamento non è stato sufficiente
  • TASK_TIMEOUT_SECONDS — timeout per singola esecuzione task (attuale: 600s); rilevante per task con file lunghi che richiedono più tempo di inferenza

Sfide Aperte

I Problemi da Risolvere

  • Task 1–4 saturano l'accuracy: expert e beginner entrambi al 100% — la differenza di ruolo non è misurabile; servono task più difficili o soglie più alte
  • Task7 expert overconfidence: confidence 1.0 su risposta sbagliata — brier score 0.33; il modello non si accorge di sbagliare
  • Retry cieco: senza reiniettare il feedback del judge nel prompt, il retry può ripetere lo stesso errore (task7 expert: 3 tentativi → wrong)
  • False positive rate non tracciato: i modelli potrebbero segnalare vulnerabilità inesistenti — non c'è ancora un task "controllo negativo"
  • Dati 1B beginner da ripulire: i run precedenti usano deepseek-r1 invece di qwen — override attivo, ma i vecchi file vanno eliminati e rieseguiti
  • Task6 full timeout: 2891 righe → latenza di prefill supera 600s su hardware locale

Roadmap

Prossimi Passi

  • Eseguire setup 1B su task5–9 con il mapping corretto (qwen2.5-coder:1.5b-base per beginner) e confrontare con i risultati 1A attuali
  • Aggiungere gemma4:e4b come agent su task6/7 — è già in uso come judge; misura il delta 2B → 4B sulla stessa rubrica
  • Retry con feedback: reiniettare il judge output nel prompt di retry — misurare se l'expert su task7 si corregge
  • Task controllo negativo: codice Go sano senza vulnerabilità — misurare false positive rate per ogni modello
  • Varianti full-file: aumentare timeout o passare a GPU per task7_full (501 r.) e task8_full (858 r.); task6_full rimane sfida aperta
  • Task più difficili per task1–4: scenari che distinguano expert da beginner — la saturazione a 100% non produce insight
