Can LLMs Find
Real 5G Vulnerabilities?

Expert vs Beginner framing · Automated rubric evaluation · Real CVEs from free5GC

Measuring LLM competence on operational tasks and security code review

Context

5G Core Networks Run on Open-Source Code

  • 5G core networks are the software stack that connects billions of devices — increasingly built on open-source projects like free5GC, written in Go
  • A CVE (Common Vulnerabilities and Exposures) is a publicly disclosed security flaw in real production code — free5GC has several, all found manually by domain experts reading thousands of lines
  • Manual review is expensive: each bug requires simultaneous knowledge of Go semantics, the Gin web framework, and 3GPP protocol specs
  • Research question: can a locally-run LLM perform a blind code review and find the same vulnerabilities — without being told what to look for?
  • And: does telling the model it is a "senior 5G engineer" vs a "junior technician" change what it finds?

What We Are Looking For

A CVE Is a Subtle Bug in Real Code

Example — UDR handler · 6 CVE advisories
// returns 404 if resource not found func GetSubscriptionData(c *gin.Context) { data, err := db.Fetch(id) if err != nil { c.String(http.StatusNotFound, err.Error()) // ← missing return } // execution continues with nil data process(data) }
In Gin, writing a 404 response does not stop execution. Without return, the handler calls process(nil) — a flaw repeated across 6 separate endpoints.
Why is this hard to find?
  • The code compiles and runs — no syntax error, no crash in normal cases
  • Requires knowing Gin-specific behavior (not standard Go)
  • Pattern repeated across multiple similar functions in a 2891-line file
  • A generic security scanner would not flag it — domain knowledge required
The LLM receives only the code — no CVE description, no patch, no hint about what to look for.

Experiment Design

Expert vs Beginner — Two Controlled Setups

Setup 1A · Control
Same Model, Different Framing
Expert and Beginner use the same underlying LLM
Any difference in results comes only from the system prompt
Isolates the pure effect of role framing
Setup 1B · Comparison
Different Models per Role
Expert → larger, more capable model
Beginner → smaller model
Measures combined effect of framing and model capability
Repetitions
Consistency Measurement
Each task runs 3 times independently (temperature 0.3)
Measures real variance — not just a single lucky answer
Up to 3 retries per attempt if the answer is wrong

Both roles produce structured Markdown output (Reasoning → Answer → Confidence) for uniform automated evaluation

Tasks

Three Categories — Increasing Difficulty

Category 1 — Deterministic
Math Tasks (1–2)
Integer and float verification
Ground truth comparison — no judge needed
Baseline: are models reliable on unambiguous tasks?
Category 2 — Textual
5G Operational (3–4)
Anomaly diagnosis, root-cause analysis
Evaluated by a judge LLM with rubric
Tests domain understanding without code
Category 3 — Security
Code Review (5–9)
Real Go code from free5GC (PCF, UDR, AMF, UDM)
Real published CVEs as ground truth
Blind review — no hints about what to look for
Each task also has a full-file variant (up to 2891 lines)

Evaluation Method

Judge LLM + Rubric-Based Scoring

  • Per-task rubric: a set of weighted criteria defining what "correct" means for each vulnerability (e.g. "identified the missing return", "located the correct function", "explained the impact")
  • Judge LLM scores the agent's response against the rubric — it receives the code and the rubric, but not the ground truth text
  • Normalized score ≥ 0.7 → verdict: correct (conservative threshold — must cover >70% of rubric criteria)
  • Rubric-based LLM-as-judge is a validated paradigm — see RUBRICEVAL (Pan et al., 2026, Fudan/Ant Group)
Example — AMF task
Rubric breakdown
vulnerability_class_score 1/1
code_location_score 2/2
missing_default_score 4/4 ← key CVE
impact_assessment_score 2/2
total 9/9 → norm 1.000 → correct
Results

What the Models Actually Did

Setup 1A · same LLM for both roles · 3 repetitions per task

Results — Tasks 1–5

Baseline: All Tasks Passed

100% accuracy across all repetitions — no differentiation between Expert and Beginner
Tasks 1–2 · math · both roles100% · deterministic
Tasks 3–4 · 5G textual · both roles100% · norm 1.000
Task 5 · PCF CORS · both roles100% · norm 1.000
  • Tasks 1–4 saturate accuracy — expert and beginner both score 100%; the framing makes no measurable difference on easy tasks
  • Task 5 (PCF CORS): AllowAllOrigins + AllowCredentials is a well-known anti-pattern — recognized by both roles on 65 lines of code with no difficulty

Results — Task 6 (UDR)

A Model Size Threshold

2B parameter model: 0/12 correct — systematic failure across all repetitions and roles
Small model (2B) — both roles
0% Accuracy
missing_return_score: 0/4 in every single run
The primary CVE (6 separate GHSA advisories) never found
Responses discuss generic Go issues — miss the Gin-specific pattern
Larger model (4B) — both roles
100% Accuracy
missing_return_score: 4/4 in all repetitions
Correctly identifies c.String(404) without return
Explains Gin execution continues after writing the response
  • Why is this hard? Requires knowing Go semantics + Gin middleware behavior + 3GPP SBI flow simultaneously — a pattern rarely documented in generic security training data
  • Scaling threshold: 1.5B → 2B fails; 4B resolves it — a clear capability boundary for this type of multi-layer reasoning

Results — Task 7 (AMF)

An Unexpected Reversal

Expert — same model
66.7%
Rep 3: 3 retries → all wrong
missing_default_score: 0/4 every attempt
Confidence stays at 1.0 throughout — overconfident on a wrong answer
Brier score: 0.33 (penalized for miscalibration)
Beginner — same model
100%
All 3 reps: correct on first attempt
missing_default_score: 4/4 in 2/3 reps
Finds: "The switch statement lacks a robust default case — if an unknown content type is provided, the switch falls through, err remains nil"
More structured, bullet-point response

Not a context window issue — the prompt difference is 8 tokens. Hypothesis: "senior expert" framing induces verbose analysis of obvious bugs, preventing systematic switch-statement scanning.

Open Problems

What Remains Uncertain

  • Is the task 7 reversal about framing or capability? Follow-up experiments planned: neutral prompt, expert with constrained style, asymmetric model setups — do any close the gap?
  • False positive rate not measured: models may report vulnerabilities that do not exist — no negative-control task (clean code) exists yet
  • Rubric contamination risk: the rubric was generated with knowledge of the CVEs — potentially circular. Domain experts need to validate whether the rubric is fair
  • Full-file performance: the 2891-line UDR file times out — is this a capacity limit or a latency problem on local hardware? Open with larger models
  • Scaling curve incomplete: 2B → 4B threshold confirmed; the next point (7B, 14B, or larger cloud model) is not yet measured

Next Steps

What We Are Doing About It

  • Domain expert validation: sending task 5, 7, 8 results to the 5G engineers who found these CVEs manually — they validate whether the reasoning is plausible and whether the rubric criteria are fair
  • Framing experiments (local): neutral prompt, expert with concise style, beginner with injected technical hint — isolating what exactly the framing changes on task 7
  • Larger models via cloud API: extending the scaling curve beyond 4B — does the expert/beginner reversal disappear with more capable models?
  • Negative control task: clean Go code without known CVEs — measuring false positive rate for each model and framing
  • Retry with judge feedback: currently retries are blind (model knows it was wrong but not why) — reinject the judge's scoring breakdown to measure if targeted feedback helps
