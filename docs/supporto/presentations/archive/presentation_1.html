Overview

Cos'è questo progetto

  • Un esperimento controllato multi-agente su scenari 5G, orchestrato con LangGraph
  • Due ruoli LLM — expert e beginner — rispondono agli stessi task e vengono confrontati
  • Due setup sperimentali (1A e 1B) permettono di isolare l'effetto del modello dal ruolo
  • Ogni task viene ripetuto più volte per misurare consistenza — non solo accuracy

Parametri di Esecuzione

Ripetizioni e Retry

Due meccanismi distinti con scopi ortogonali
  • REPETITIONS — ripete lo stesso task N volte per misurare consistenza; con temperatura > 0 misura varianza reale tra esecuzioni indipendenti
  • MAX_RETRIES — tentativi aggiuntivi dentro una singola ripetizione, attivati solo se il verdict è wrong
  • Limite attuale: il feedback del judge non viene reiniettato nel prompt di retry — l'agente non sa perché ha sbagliato
  • Temperatura 0: se la temperatura è 0, le ripetizioni misurano stabilità del parsing e dell'infrastruttura — non varianza LLM

Output

Cosa viene salvato

  • Per ogni ripetizione: due file in results/<experiment_id>/<role>/
    • <task_id>_repN.json — risultato completo: verdict, history, judge_score, elapsed_seconds
    • <task_id>_repN_solution.json — snapshot della ground truth al momento del run
  • Report aggregati in results/evaluation/ al termine di ogni run
    • scores_1A.md e scores_1B.md — accuracy, confidence media, breakdown per task, consistenza reasoning
    • comparison.md — confronto sintetico 1A vs 1B
    • consistency.md — differenze tra ripetizioni quando il JSON finale cambia
  • Grafo LangGraph: python main.py --export-graph docs/graph.png

Confronto

Cosa misura ogni metrica

Accuracy
Correttezza
Cosa: % verdict correct per ruolo/setup/task
Math: deterministico, oggettivo
Textual: dipende dal judge e dalla rubrica
Rischio: satura a 100% se i task sono troppo facili
Confidence
Calibrazione
Cosa: media di confidence dichiarata dall'agente
Attuale: salvata ma non incide sul verdict
Futuro: penalità overconfidence, score composito
Nota: misura auto-percezione, non calibrazione reale
Consistenza
Stabilità
Cosa: variazione JSON tra ripetizioni dello stesso task
Attuale: confronto stringente (equality)
Limite: segnala diff anche a parità di senso
Futuro: confronto semantico via embedding

Sfide Aperte

I problemi da risolvere

  • Judge come fonte di verità: se la rubrica è ambigua il judge premia risposte plausibili — non necessariamente corrette
  • Retry cieco: senza reiniettare il feedback del judge nel prompt, il retry ripete lo stesso errore
  • Task troppo facili: se l'accuracy satura, le differenze expert/beginner e 1A/1B diventano invisibili
  • Token e costi non tracciati: elapsed_seconds cattura la latenza ma non i token per chiamata — confrontare costo/qualità è difficile
  • Lingua dei prompt: prompt in italiano — A/B test IT vs EN non ancora eseguito
