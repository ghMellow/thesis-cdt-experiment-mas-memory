Evaluation Method

Judge LLM + Rubric-Based Scoring

  • Separate model: judge is a different LLM. It receives the task scenario, the agent response, and the rubric, but never the ground truth text
  • Rubric categories: weighted criteria per task: vulnerability class, code location, security impact, fix quality
  • Threshold: normalized total ≥ 0.7 → verdict correct
  • Methodology: RubricEval (Pan et al., 2026) validates rubric-based LLM-as-judge at scale
  • Agent vs judge: the agent receives only source code and its role framing, no rubric and no expected answers. The judge is guided by the rubric by design, but since the rubric was derived from the same published patches, a potential circularity exists and is flagged for external validation.

Domain

free5GC · Real Advisories as Ground Truth

free5GC is an open-source Go implementation of a complete 5G Core network, used in academic research and early deployments.
  • GHSA (GitHub Security Advisory): GitHub’s advisory format. Each entry includes the patch diff, affected versions, and a link to the fix commit. Written by project maintainers, these are the ground truth for our detection experiment.
  • Invisible to the compiler: none of these bugs are caught by go vet or standard static analysis. A missing return in Gin, an empty switch default, a missing validator call — all syntactically valid Go. Detecting them requires knowing how Gin handles control flow, how 3GPP defines valid inputs, and where the 5G Core trusts caller data.

Setup

Task and Rubric Construction

Tasks and rubrics were built using Claude Sonnet 4.6 (cloud), which read the real CVE patches and the free5GC source code to produce task excerpts and scoring criteria.
  • Task excerpts: full file version and a shorter selection of the most relevant sections. No code was rewritten, only selected.
  • Rubrics: per-task weighted criteria derived from the published GHSA patches. What a competent reviewer should identify, at what granularity, and with what score weight.
  • Asymmetry by design: a powerful cloud model sets the problem; local models run inference on consumer hardware. This separation is intentional and itself becomes a variable to test.

Findings · Framing

The Expert Paradox

On task7, Beginner outperforms Expert (same model). Expert finds real bugs and earns partial points but never reaches the switch statement — highest-weighted criterion (4/9).
  • Not length (F6): prompts differ by ~8 tokens
  • Not verbosity (F17): "one sentence per finding" makes it worse — verbosity is part of the reasoning chain
  • Not a single instruction (F18): "scan switch statements" hint recovers 66.7%, not 100%
  • No framing is not the answer (A1): removing role drops Beginner 100% → 33.3%, Expert unchanged
  • Capacity-dependent (F19/F20/F21): scaling expert-only: e2b 0.0% → e4b 66.7% → 31b 100%. The paradox is a framing × capacity interaction, not a structural expert limitation
Open question: framing shapes attention style, not knowledge. Scaling resolves the paradox (31b expert = 100%) but does not answer what minimum context a CDT needs without cloud access.

Towards CDT

Tasks and rubrics were built with Claude 4.6 (cloud). In an autonomous CDT the system must define its own evaluation criteria from observed anomalies, using only local resources. Can a small model replace the cloud model in the setup phase?
Long-term goal: a CDT that continuously monitors a 5G cell, autonomously identifies anomaly patterns, formulates its own evaluation criteria, and evolves its detection capability without external supervision.
CVE Deep Dive — task5 & task6

CORS Misconfiguration and the Gin Control Flow Bug

task5 · PCF · GHSA-98cp
Bug: AllowAllOrigins + AllowCredentials simultaneously enabled in Gin CORS middleware
s.router.Use(cors.New(cors.Config{ AllowMethods: []string{"GET", "POST", ...}, AllowAllOrigins: true, AllowCredentials: true, // ← CVE MaxAge: CorsConfigMaxAge, }))

Any origin can trigger credentialed requests — cookies, tokens, and auth headers exposed cross-origin. A well-known anti-pattern.

task6 · UDR · 6× GHSA-wrwh
Bug: missing return after writing a 404 response — execution continues with nil data
func GetSubscriptionData(c *gin.Context) { data, err := db.Fetch(id) if err != nil { c.String(http.StatusNotFound, err.Error()) // ← missing return } // execution continues with nil data process(data) }

In Gin, writing a response does not halt execution. Pattern repeated across 6 endpoints. Requires Gin-specific knowledge to catch.

Open Questions

What Remains Uncertain

  • Framing paradox — task 7: is the Expert/Beginner reversal reproducible? Planned ablations: neutral prompt, constrained expert style, beginner with technical hint
  • Rubric contamination: rubric was built with CVE patch access — does the judge evaluate genuine understanding, or echo rubric language derived from the patches?
  • False positive rate: not yet measured — no negative-control tasks (clean Go handlers without CVEs) exist yet
  • Task 6 hint: “Pay special attention to…” means task 6 measures pointer-guided analysis, not blind detection — results must be interpreted separately
  • Scaling curve: 2B → 4B threshold confirmed; 12B, 32B, large cloud models not yet tested — does the framing paradox persist at higher capacity?
Summary

2B fails task 6. 4B solves it.
Beginner beats Expert on task 7.

1. Capacity threshold is real

Model size sets a sharp boundary for Gin-specific vulnerability detection. 2B fails systematically; 4B resolves it entirely. Framing does not close this gap.

2. Framing can hurt

“Senior engineer” framing induces verbose analysis that misses simpler bugs — a testable hypothesis requiring systematic ablation experiments.

3. Rubric works, needs validation

Judge + rubric produces consistent, discriminating scores. External 5G domain experts must confirm fairness and rule out circularity from CVE patch access.

The systematic failure is a result. The reversal is a hypothesis. The validation package is ready.
