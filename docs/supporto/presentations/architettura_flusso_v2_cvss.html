Il flusso completo: un prompt, due rami di valutazione

Un unico prompt entra nell'agente. La risposta esce già divisa in due parti: il testo va al giudice (ramo A, LLM), la stima CVSS va allo script (ramo B, zero LLM). I due rami non si influenzano mai e confluiscono nello stesso file JSON.

Blocco 1 — Il prompt unico e l'agente

Tutto in una sola chiamata: task, contesto NF e istruzioni CVSS. L'agente non sa di essere valutato in due modi.

Blocco 1 · Dettaglio

Cosa contiene il prompt (nell'ordine)

  • 1 · Il task: scenario + codice Go di una NF free5GC, richiesta di security review cieca (docs/tasks/<task>.md)
  • 2 · Hint di contesto NF (attivabile, CVSS_CONTEXT_HINT_ENABLED): un paragrafo che dice all'agente che il codice gira in un core 5G con OAuth2/mTLS sulla SBI — serve per stimare l'impatto in modo realistico
  • 3 · Istruzioni CVSS (utils/cvss_utils.py): per ogni vulnerabilità trovata, emettere function / vector / score. Il vettore richiesto copre tutte le 11 metriche base (anche SC/SI/SA, impatto sui sistemi a valle). Include la legenda completa dei valori CVSS 4.0, mai i valori corretti
  • Il file del task su disco non viene toccato: hint e istruzioni sono iniettati a runtime
Output dell'agente (un'unica risposta)
### Answer La review testuale: vulnerabilità, dove, perché ... ### Reasoning / ### Confidence ... 0.8 ### CVSS Estimate ← la parte nuova function: HTTPQueryAmData vector: CVSS:4.0/AV:N/AC:L/AT:N/PR:L/UI:N /VC:H/VI:N/VA:N/SC:N/SI:N/SA:N score: 6.9 function: HTTPModifyAmData vector: ... score: ...

Ramo A — Il giudice a rubrica (come in v1)

Valuta solo il testo. Non vede mai la stima CVSS né la ground truth.

Ramo A · Dettaglio

Giudice: valuta il testo, decide i retry

  • Input: scenario + rubrica pesata + risposta testuale dell'agente. La sezione ### CVSS Estimate non gli viene passata
  • Verdetto: somma punteggi per criterio → normalizza → correct se ≥ 0.7
  • Retry: se wrong e restano tentativi, l'agente riprova vedendo solo la propria risposta precedente (mai feedback, punteggio o soluzione)
  • Ruolo nel sistema: automazione interna — evita la validazione manuale di ogni output e dà all'agente una chance di auto-correzione. È un dettaglio della "scatola nera": chi consuma gli output può ignorarlo
Perché tenerlo, in ottica articolo: il framework ad agenti con validatore è lo standard corrente per ridurre allucinazioni con modelli locali/limitati. Ma il sistema funziona anche bypassandolo: il ramo CVSS non dipende in alcun modo dal giudice.
Modello del giudice: voce separata MODELS["judge"] — può essere lo stesso dell'agente o uno diverso. Se coincide, c'è un bias di stile noto: da giustificare nella run definitiva.

Ramo B — Lo script CVSS (deterministico)

Zero LLM: puro Python che confronta la stima con la ground truth. Non influenza mai il verdetto del giudice.

Ramo B · Dettaglio 1/4

Il riferimento: la ground truth delle CVE reali

  • Dataset: File_Free5gc_Vulnerabili/cve_metrics_normalized.json — le CVE 2026 di free5GC, una entry per CVE, mappata sul task che contiene quel codice
  • Score e vettore sono quelli ufficiali pubblicati (NVD o CNA, campo cvss_source): il nostro codice non calcola nulla lato ground truth
  • score_type B vs BT: per 3 CVE lo score pubblicato include la metrica Threat E (BT, più basso); per quelle il dataset ha anche base_score_B, il base puro ricalcolato una tantum con la libreria Python cvss
  • handler_functions: i nomi delle funzioni vulnerabili — sono la chiave con cui lo script aggancia i finding dell'agente alle CVE
Una entry del dataset (struttura semplificata)
{ "id": "CVE-2026-…", "task_id": "task8_vuln_udm", "handler_functions": ["HTTPQueryAmData", …], "in_task_excerpt": true, "cvss_source": "NVD", ← fonte ufficiale "cvss": { "base_score": 8.7, ← pubblicato "score_type": "B", ← B o BT "base_score_B": null, ← solo se BT "vector": "CVSS:4.0/AV:N/…/VC:H/VI:N/VA:N/…", "base_metrics": { "AV":"N", "VC":"H", … } } }
Nota metodologica: in CVSS 4.0 lo score non è una somma di pesi per lettera — deriva da macrovettori + lookup table ufficiale FIRST. Lo script usa proprio quell'algoritmo (libreria cvss) per ricalcolare lo score dal vettore stimato — validato ricalcolando i 10 vettori della ground truth: tutti coincidono con gli score NVD/CNA pubblicati.
Ramo B · Dettaglio 2/4

Passo 1 — Abbinare i finding alle CVE (l'incipit di tutto)

1Caso semplice · la maggior parte dei task
1 CVE nel file → abbinamento diretto
Una sola CVE attesa, un finding dell'agente: quel finding è la stima per quella CVE. Nessun matching necessario
2Caso multi-CVE · es. task6 (UDR) — il cuore del passo 1
Più CVE nello stesso file → match sul nome della funzione
Qui il problema esiste davvero: l'agente riporta più finding, il file ha più CVE — quale stima confronto con quale CVE?
La chiave è il campo function che l'agente compila per ogni finding: il nome della funzione Go che gestisce la richiesta (handler), confrontato con le handler_functions registrate nella ground truth di ogni CVE
Match per sottostringa, case-insensitive: "HTTPQueryAmData handler" o "httpqueryamdata" agganciano comunque
2l'esempio del match sul nome funzione
# finding dell'agente # ground truth (CVE) function: "HTTPQueryAmData" ↔ handler_functions: vector: CVSS:4.0/… ["HTTPQueryAmData"] score: 6.9 → abbinato a quella CVE
3I casi residui
Finding senza CVE corrispondente: contati a parte (unmatched_findings), mai penalizzati
CVE attese che nessun finding copre → missed_cves
Il "non penalizzare i finding extra" è una scelta di design: sono esattamente il caso d'uso che interessa a voi — potenziali vulnerabilità senza CVE associata.
Ramo B · Dettaglio 3/4

Passi 2–4 — Come valutiamo (aggiornato post call 11)

2Passo 2 · Score a fasce (banda)
Quanto ci azzecca il numero
Δ = |score stimato − score di riferimento|, poi punti a fasce:
Δ ≤ 0.5 → 3 pt  ·  Δ ≤ 1.5 → 2 pt  ·  Δ ≤ 3.0 → 1 pt  ·  oltre → 0 pt
Calcolato due volte: vs score pubblicato ("banda vs pubblicato") e vs base B puro ("banda vs base B")
Esempio: stimato 6.9 vs pubblicato 8.7 → Δ = 1.8 → 1 pt
Le soglie (0.5 / 1.5 / 3.0) sono provvisorie, decise in call 10 e centralizzate in config.CVSS_SCORE_BANDS: ricalibrarle è un cambio di una riga, i dati grezzi restano invariati.
3Passo 3 · Vettore campo per campo
Quanto ci azzecca con le lettere
Exploitability (AV/AC/AT/PR/UI): conta i campi uguali, 0–5 · Impatto (VC/VI/VA): 0–3 — è l'impatto che discrimina su questo dataset
Oltre al match binario, distanza di severità ordinale [0–1] per gruppo: sbagliare di un livello (H vs L) non pesa più come sbagliare all'estremo opposto (H vs N)
SC/SI/SA (sistemi a valle): valutate a parte, solo sulle run in cui l'agente le emette
Come appare nei report (campi divergenti in evidenza)
| CVE-2026-… — rep 3 | stimato | pubblicato | |--------------------------|---------|------------| | AV — Attack Vector | N | N | | PR — Privileges Required | L | L | | VC — Confidentiality | *L* | *H* | | VI — Integrity | N | N |
Passo 4 · Matematica ufficiale (novità post call 11): il vettore stimato viene ripassato nell'algoritmo FIRST 4.0 → ne esce lo score che quel vettore vale davvero. Da qui: coerenza interna (lo score dichiarato dall'agente combacia con il suo stesso vettore?) e distanza in spazio score (score ricalcolato vs base B ufficiale). Caso reale: agente dichiara 5.1, il suo vettore vale 7.1 — molto più vicino all'8.7 pubblicato di quanto sembrasse.
Ramo B · Dettaglio 4/4

Dove servono le vostre idee

Il punto chiave per voi: tutto ciò che serve per provare criteri alternativi è già salvato nei JSON (vettore stimato, vettore pubblicato, entrambi gli score, nomi funzione). Cambiare la metrica di valutazione non richiede rilanciare le run: si riprocessano i risultati esistenti.
Già implementato (post call 11)
✓Distanza sul vettore: matematica ufficiale 4.0 (algoritmo FIRST, libreria cvss) + distanza di severità ordinale per campo
✓Coerenza interna: lo score dichiarato viene confrontato con lo score che il vettore stimato produce davvero
Temi ancora aperti · in ordine di impatto
1B vs BT: quale riferimento usare come metrica principale nell'articolo
2Calibrazione bande: le soglie attuali sono ragionevoli per il triage reale?
3Matching multi-CVE: il criterio "nome funzione" basta, o serve altro (riga di codice, tipo di bug)?
4Interpolazione tra vettori: la distanza FIRST vettore↔vettore vera e propria (materiale Mariano) — oggi la distanza in spazio score usa i due score ricalcolati
Come si procede (già successo con la matematica ufficiale): la metrica nuova si implementa nello script (utils/cvss_eval.py) e si ricalcola tutto retroattivamente con python -m utils.cvss_eval — le run 1–3 hanno già i nuovi campi, zero run rilanciate.

Blocco finale — Output e punti di uscita

Tutto confluisce in un JSON per run. Da lì i dati si estraggono e si usano come si vuole.

Output · Dettaglio

Il JSON di risultato: dove sta cosa

{ "task_id": "task8_vuln_udm", "model": ..., "repetitions": [ { "final_answer": { "answer": ..., "reasoning": ..., "cvss_estimate": { ← stima grezza "findings": [ {"function", "vector", "score"} ] } }, "verdict": "correct", ← ramo A (giudice) "judge_score": { ... }, "cvss_eval": { ← ramo B (script) "matched": [ { "cve_id", "estimated_vector", "published_vector", "score_band_vs_published", "impact_match", ... } ], "missed_cves": [...], "aggregates": {...} } } ] }
  • final_answer.cvss_estimate — ciò che il modello ha detto, così com'è: la lista di finding con vettore e score
  • verdict / judge_score — l'esito del ramo A (rubrica)
  • cvss_eval — l'esito del ramo B: matching, bande, vettori stimato vs pubblicato affiancati
  • Schema completo documentato in docs/results_reference/schema_textual.json, con un campo overview che spiega come leggere il file
Output · Per chi consuma i dati

Punti di uscita: prendere i vettori e usarli come volete

  • Uscita 1 — le stime grezze: ogni cvss_estimate.findings è già una lista JSON di vulnerabilità con vettore e score. Si estrae con due righe e si ordina, filtra, esporta come si vuole (es. ranking decrescente per triage manuale — proposta Lorenzo)
  • Uscita 2 — il confronto con la GT: cvss_eval dà il quadro già valutato (matched/missed, bande, campi divergenti), utile per le metriche dell'articolo
  • Uscita 3 — i report leggibili: tabelle Markdown per-CVE con vettore stimato vs pubblicato affiancati, campi divergenti in grassetto (results/evaluation/)
  • Nessuna uscita richiede il giudice: la scatola nera resta "codice dentro → lista di vulnerabilità quantificate fuori"
Esempio: tutti i vettori stimati, ordinati per score
import json, glob rows = [] for f in glob.glob("results/*/*/*/*.json"): data = json.load(open(f)) for rep in data["repetitions"]: est = rep["final_answer"].get("cvss_estimate") or {} rows += est.get("findings", []) rows.sort(key=lambda r: r.get("score", 0), reverse=True) # → lista pronta: [{"function", "vector", "score"}, ...]
Questo è il punto chiave per l'esperimento 3 (SonarQube + LLM): stesso formato di output, cambia solo l'input.
Riepilogo v2

Il sistema con CVSS in tre frasi

1 · Un prompt
Un solo agente riceve task + contesto NF + istruzioni CVSS in una chiamata unica, e risponde con review testuale e lista di finding quantificati (vettore + score).
2 · Due valutazioni indipendenti
Ramo A: giudice LLM a rubrica sul testo (decide verdetto e retry). Ramo B: script Python deterministico sui vettori CVSS. Non si parlano mai.
3 · Output riusabile
I vettori escono come lista JSON pronta all'uso: ranking, triage manuale, metriche per l'articolo, input per l'esperimento SonarQube+LLM.

Prossimi passi sul ramo B: matematica ufficiale CVSS 4.0 (macrovettori + lookup) al posto del confronto per lettera.
