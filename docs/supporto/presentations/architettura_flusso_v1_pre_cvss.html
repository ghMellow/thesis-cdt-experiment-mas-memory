Il flusso prima del CVSS

Come funziona il sistema originale: ruoli expert/beginner, setup 1A/1B, giudice a rubrica, ripetizioni e retry. Prima l'immagine dall'alto, poi ogni blocco nel dettaglio.

Livello 0 · Vista dall'alto

Il flusso completo, in una riga

Per ogni combinazione (setup, ruolo, task) il sistema esegue N ripetizioni indipendenti. Dentro ogni ripetizione: si costruisce il prompt, l'agente risponde, il giudice valuta con la rubrica; se il verdetto è wrong l'agente ritenta (max 3 volte), poi si salva tutto in JSON. Nelle prossime slide entriamo in ogni blocco, uno alla volta.

Blocco 1 — Configurazione: cosa entra nel sistema

Prima di lanciare qualsiasi cosa, il sistema decide: quale setup, quale ruolo, quale task, quante ripetizioni.

Blocco 1 · Dettaglio

Ruoli e setup: le due leve dell'esperimento

Ruoli (framing)
Expert vs Beginner
Due system prompt diversi (agents/prompts.py): ingegnere 5G senior vs tecnico junior
Stesso task, stesso codice: cambia solo chi il modello crede di essere
Domanda: il framing cambia cosa trova il modello?
Setup 1A
Stesso modello per entrambi
Expert e beginner usano lo stesso LLM
Isola l'effetto del solo prompt di ruolo (controllo)
Setup 1B
Modelli diversi per ruolo
Expert e beginner possono usare LLM diversi (mapping in config.py → MODELS)
Misura se conta più la capacità del modello o il framing
In totale: 2 setup × 2 ruoli × N task × N ripetizioni. Ogni combinazione produce un file di risultato separato — nulla viene mai mescolato.
Blocco 1 · Dettaglio

Gli iperparametri (config.py)

  • TEMPERATURE = 0.3 — condivisa da agente e giudice. Tenuta > 0 apposta: così le ripetizioni misurano la varianza reale del modello, non solo la stabilità del parsing
  • REPETITIONS = 3 — lo stesso identico prompt viene lanciato 3 volte indipendenti, per distinguere effetti reali da rumore di campionamento
  • MAX_RETRIES = 3 — tentativi dentro una singola ripetizione, solo se il giudice dice wrong
  • TEXTUAL_PASS_RATIO = 0.7 — soglia sul punteggio normalizzato della rubrica per il verdetto correct
# config.py — i quattro numeri chiave TEMPERATURE = 0.3 # > 0: varianza reale REPETITIONS = 3 # run indipendenti MAX_RETRIES = 3 # dentro 1 ripetizione TEXTUAL_PASS_RATIO = 0.7 # soglia correct/wrong MODELS = { "expert_1A": ..., # un modello per ruolo "beginner_1A": ..., "expert_1B": ..., "beginner_1B": ..., "judge": ..., # modello del giudice }
Ripetizioni ≠ retry: la ripetizione è una nuova run da zero; il retry è un secondo tentativo dentro la stessa run dopo un verdetto wrong.

Blocco 2 — Prompt e Agente

Il cuore: come si costruisce il prompt e cosa produce l'agente.

Blocco 2 · Dettaglio

Un prompt, una risposta strutturata

  • System prompt = il ruolo (expert o beginner). È l'unica cosa che distingue i due agenti
  • User prompt = il testo del task (docs/tasks/<task>.md): scenario + codice Go di free5GC, review cieca — nessun suggerimento sulla soluzione
  • Output richiesto: Markdown con tre sezioni fisse, parsate dal codice (agents/agent_runner.py)
  • L'agente non vede mai: la rubrica, la ground truth, il feedback del giudice
Formato di output dell'agente
### Answer La vulnerabilità principale è ... ### Reasoning Ho analizzato gli handler e ... ### Confidence 0.8
Le tre sezioni vengono convertite in JSON e salvate in history[] (ogni tentativo) e final_answer (l'ultimo).

Blocco 3 — Giudice e retry

Un secondo LLM valuta la risposta con una rubrica. Se dice wrong, l'agente ritenta.

Blocco 3 · Dettaglio

Il giudice: rubrica, non ground truth

  • Cosa riceve: scenario del task + rubrica (criteri pesati, derivati dalle patch GHSA pubblicate) + risposta dell'agente
  • Cosa NON riceve: la ground truth testuale — la rubrica è la definizione operativa di "corretto"
  • Verdetto: somma i punteggi per criterio, normalizza su total_max; correct se ≥ 0.7
  • Modello del giudice: definito a parte in MODELS["judge"] — può coincidere o meno con quello dell'agente (bias noto, discusso col team)
Payload del giudice
## Scenario (testo del task) ## Rubric Total max score: 9 - missing_default_score (0-4) - impact_assessment_score (0-2) - ... ## Agent Response ### Answer / ### Reasoning / ### Confidence
Blocco 3 · Dettaglio

Il ciclo di retry: cosa rivede l'agente

Quando scatta
verdict = wrong & tentativi < 3
Il giudice boccia la risposta e restano tentativi disponibili
Il flusso torna a run_agent dentro la stessa ripetizione
Cosa vede l'agente al retry
Solo la propria risposta precedente
Il task + la sua answer/reasoning del tentativo prima + un invito neutro a riprovare
Mai: il feedback del giudice, il punteggio, la ground truth
Quando si chiude
correct, oppure 3 tentativi esauriti
Si salva l'ultimo tentativo come final_answer con il suo verdetto
Tutti i tentativi restano tracciati in history[]
Il retry è volutamente "cieco": misura se il modello riesce ad auto-correggersi senza essere guidato verso la soluzione. Ogni forma di hint invaliderebbe l'esperimento.

Blocco 4 — Salvataggio e report

Cosa esce dal sistema e dove trovarlo.

Blocco 4 · Dettaglio

Output: JSON grezzi + report aggregati

  • JSON per combinazione in results/<task>/<setup>/<ruolo>/: run_config + array repetitions con tutti i tentativi, verdetti, punteggi, tempi e token
  • Report Markdown generati a fine run (results/evaluation/): scores_1A.md, scores_1B.md, comparison.md, consistency.md
  • Metriche aggregate per ruolo: accuracy, confidence media, brier score, tentativi medi, punteggio rubrica normalizzato
  • Anomalie evidenziate: verdetti wrong, retry scattati, reasoning incoerente tra ripetizioni
results/ ├─ task7_vuln_amf/ │ ├─ 1A/ │ │ ├─ expert/<modello>.json │ │ └─ beginner/<modello>.json │ └─ 1B/ ... └─ evaluation/ ├─ scores_1A.md # tabella per ruolo ├─ scores_1B.md ├─ comparison.md # 1A vs 1B └─ consistency.md # differenze tra rep
Riepilogo v1

Il sistema pre-CVSS in tre frasi

1 · Chi risponde
Un agente LLM con framing di ruolo (expert/beginner), su codice free5GC reale, in review cieca. Setup 1A/1B decide se i ruoli condividono il modello.
2 · Chi valuta
Un giudice LLM separato con rubrica pesata (mai la ground truth). Verdetto wrong → fino a 3 retry ciechi dentro la stessa ripetizione.
3 · Come si misura
3 ripetizioni indipendenti a temperatura 0.3 per distinguere effetti reali dal rumore. Tutto tracciato in JSON + report aggregati.
